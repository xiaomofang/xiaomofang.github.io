Temperature=0 不等于确定性：从解码策略到 Batch Invariance

很多工程直觉会把 temperature=0 和“输出完全一致”画上等号。这个等号并不成立： temperature=0 固定的是解码策略，而不是生成 logits 的整个计算过程。

在实际推理服务中，即使模型、prompt、请求参数完全相同，并将 temperature 设置为 0， 多次请求仍然可能得到不同结果。原因并不神秘：如果参与 argmax 的 logits 本身出现细微差异， 那么解码规则再确定，最终 token 也可能发生变化。

近年来关于推理确定性的研究不断指向同一个结论：确定性不是一个简单的采样参数问题， 而是一项跨越模型、算子、推理框架和在线服务系统的工程属性。

确定性从哪里开始失效

从系统角度看，temperature=0 只是固定了解码策略。真正决定输出能否复现的， 是整个推理链路能否保持一致的数值计算过程。

很多人的潜意识推理链路是： 

输入固定
  ↓
logits 固定
  ↓
输出固定

但 temperature 实际上只能保证后一段关系： 

logits 固定
  ↓
argmax 固定
  ↓
输出固定

至于 logits 是否固定，并不由 temperature 决定。这正是整个问题的入口。

一、Temperature=0 固定的是什么

Temperature 作用于 logits 到概率分布的转换过程。temperature 较高时，概率分布会被拉平， 低概率 token 更容易被采样到；temperature 较低时，分布会变得更加尖锐，高概率 token 的优势被进一步放大。

当 temperature=0 时，工程实现通常不再进行随机采样，而是直接选择当前 logit 最大的 token： 

next_token = argmax(logits)

因此，temperature=0 固定的是 Decoding Policy，也就是“如何从当前 logits 中选择下一个 token”。 它并没有固定 Model Computation，也就是“这些 logits 是如何被计算出来的”。

关键区别： temperature=0 消除的是采样随机性；它不保证 logits 的计算过程逐 bit 一致。

二、为什么相同 Prompt 仍然可能产生不同输出

假设某一步生成时，两个候选 token 的 logits 非常接近： 

Token A : 8.000001
Token B : 8.000000

Greedy decoding 会选择 Token A。但如果由于不同数值计算路径产生了极小误差： 

Token A : 7.999999
Token B : 8.000000

argmax 的结果就会变成 Token B。对于整体模型而言，8.000001 和 7.999999 几乎没有语义上的差别； 但对于 argmax 来说，第一名和第二名交换了位置，结果就完全不同。

Transformer 又是自回归模型。当前 token 会进入下一轮计算，成为后续上下文的一部分。 因此一次局部分叉可能被逐步放大，最终表现为整段回答的差异。一个看似微不足道的数值扰动， 在生成系统中可能成为两条输出路径的分岔点。

三、问题不只是浮点误差，而是 Batch Invariance

这种现象常被简单归因于“GPU 并行计算存在浮点误差”。这个解释并不完全错误，但还不够准确。 浮点加法确实不满足结合律，例如： 

(1e30 + -1e30) + 1
1e30 + (-1e30 + 1)

在有限精度表示下，两种计算顺序可能得到不同结果。但现代推理框架中的大部分算子， 在固定输入、固定 kernel 和固定执行形状下，通常能够保持 bitwise deterministic。

真正值得关注的问题是 Batch Invariance：同一个请求单独执行时得到的结果， 与它被放入另一个 batch 中执行时得到的结果，未必完全一致。

这不是“GPU 随机”这么简单，而是高性能推理系统为了吞吐，对不同输入形状选择了不同计算路径。

四、Batch Shape 如何影响结果

Transformer 中大量计算都包含 reduction 操作，例如 matrix multiplication、attention、RMSNorm 和 softmax。 这些算子需要将大量中间结果归约成最终输出。

为了获得更高吞吐，高性能 kernel 会根据输入形状动态选择分块策略、并行方式、reduction 路径和 warp 调度方案。 因此，同一个请求如果处于不同 batch 中，就可能经过不同的数值计算路径。 

第一次请求：
Batch = [A]

第二次请求：
Batch = [A, B, C, D]

对推理引擎来说，这已经是两个不同的计算场景。kernel 的执行策略可能发生变化，reduction 的顺序也可能发生变化， 最终导致 logits 出现末位差异。

五、Dynamic Batching 是在线服务中的放大器

在线推理服务为了提高吞吐，通常都会采用 dynamic batching：将不同用户、不同长度、不同到达时间的请求， 动态拼接到同一个 batch 中。

从用户视角看，两次请求完全相同；但从服务端视角看，它们可能处于完全不同的 batch 环境中。 这意味着，相同请求并不一定对应相同计算过程。服务器负载、请求到达顺序、序列长度分布等因素， 都可能间接影响最终结果。

推理系统中的不确定性，往往来源于服务系统本身，而不仅仅来源于模型或采样参数。

六、MoE 模型会进一步放大这种现象

对 dense 模型而言，数值扰动主要体现在 logits 的末位变化；但对 MoE（Mixture of Experts）模型来说， 影响可能更大。MoE 中的 router 需要根据得分选择 top-k expert。 

Expert 3 : 0.500001
Expert 4 : 0.500000

如果出现极小扰动： 

Expert 3 : 0.499999
Expert 4 : 0.500000

路由结果就会发生变化。此时改变的不仅是某个 token 的概率，而是后续参与计算的 expert 本身。 因此，MoE 模型往往比 dense 模型对数值路径更加敏感。

七、为什么这件事会影响 RL 训练

如果只是面向用户的聊天应用，偶发的措辞差异通常可以接受。但在 on-policy reinforcement learning 中， 问题会变得更加敏感。on-policy RL 的核心假设之一是：用于采样数据的策略，与用于更新参数的策略保持一致。 

π_sample ≈ π_train

现实系统中，采样侧可能运行在 vLLM、SGLang、TensorRT-LLM 等推理引擎上； 训练侧可能运行在 FSDP、Megatron-LM、DeepSpeed 等训练框架上。即使权重完全相同， 两套系统仍可能因为 kernel 实现、并行策略、reduction 路径和数值精度产生细微差异。

对普通生成任务而言，这种差异可能只是措辞变化；但对 RL 训练而言，它意味着采样策略与训练策略不再严格一致。 这种偏差会进一步影响 advantage estimation、KL constraint 和 policy update，最终影响训练稳定性。

八、如果线上需要复现，应该怎么设计

对闭源 API 来说，seed 通常只是 best-effort 保证。更稳妥的做法是记录完整请求上下文： model version、prompt、temperature、top-p、max tokens、seed 和 system fingerprint。

同时需要认识到：即使 seed 和 fingerprint 完全一致，长文本生成也未必能够严格复现。 生成越长，token 决策次数越多，出现分叉的概率也越高。

场景 建议
普通业务 固定模型版本、prompt 和请求参数。
回归测试 保存 token 序列、logprobs、请求参数和评测数据版本。
审计场景 明确定义允许漂移范围，并记录 system fingerprint。
强确定性场景 使用 deterministic kernel、fixed reduction order 和 batch-invariant operators。

如果业务对复现性要求极高，就需要在推理框架层面保证 deterministic kernel、固定 reduction 顺序和 batch-invariant operators， 并接受一定吞吐损失。

结论：Temperature=0 并不等于严格确定性

Temperature=0 能够消除采样带来的随机性，但它无法保证 logits 的计算过程完全一致。 在现代推理系统中，浮点归约路径、kernel 实现、dynamic batching、并行通信、MoE 路由以及服务端配置变化， 都可能导致 logits 出现细微差异。

当这些差异恰好发生在概率接近的候选 token 上时，argmax 的结果就可能改变。 因此，真正的确定性不是一个采样参数，而是一项系统属性。

它不仅依赖解码策略，还依赖算子实现、推理框架、调度策略以及整个服务链路是否能够保持一致的数值计算路径。 理解这一点，才能真正理解：为什么同一个 prompt、同一个模型、同一个 temperature，仍然可能得到不同答案。

Temperature=0 不是确定性的终点，而是理解推理系统从哪一层开始失去确定性的起点。
