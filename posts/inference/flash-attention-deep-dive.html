FlashAttention 深度剖析：从存储层级到 IO-Aware Exact Attention

FlashAttention 没有改变 attention 的数学定义，却改变了数据在 HBM 与片上 SRAM 之间的流动方式。 理解它，需要先理解 GPU 的存储层级，再理解为什么标准 attention 是 memory-bound。

论文标题是 FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness。 四个关键词分别对应四个问题：快、省显存、结果精确、对硬件 IO 有意识。 本文的目标，是像从第一性原理推导一样，把这条思路完整走一遍—— 读完你应该能回答：为什么 attention 慢？FlashAttention 改了什么？V1 到 V4 各自在优化什么？

一、先搞清存储层级：快、小、贵 vs 慢、大、便宜

计算机存储不是一块铁板，而是分层金字塔。越靠近计算单元，越快、越小、越贵； 越远离计算单元，越慢、越大、越便宜。FlashAttention 的全部设计，都建立在这个事实上。

CPU 侧：L1、L2、L3

L1 cache 最靠近 CPU 核心，容量通常几十 KB/核，延迟最低。 L2 cache 更大一些，仍多为每核私有，承接 L1 miss。 L3 cache 再上一层，多核共享，用来减少访问主存 DDR 的次数。 三者本质上都是 cache——用 SRAM 缓存热数据，不是持久存储。

GPU 侧：Register、Shared Memory、HBM

GPU 上每个线程有极小的 Register；每个 SM 有 Shared Memory（片上 SRAM，~192KB/SM 量级）， 程序员可以显式控制数据布局。再往外是 L2 cache，最外层是 HBM（High Bandwidth Memory，GPU 显存）。

以 A100 为例：HBM 约 40–80 GB，带宽 ~1.5–2 TB/s；每个 SM 的 SRAM 约 192 KB，有效带宽可达 HBM 的 ~10 倍。 算力（FLOPS）增长比内存带宽快得多——软件必须主动适配这种失衡，这就是 IO-aware 的硬件背景。

层级 典型容量 角色
Register / SRAM KB 级 FlashAttention tile 的工作区；online softmax 统计量也在这里维护。
HBM 几十 GB 存放 Q、K、V、O 全量张量；标准 attention 还会写入 N×N 的 S、P。
SSD TB 级 模型权重、checkpoint；推理前加载到 HBM，不参与 attention kernel 内部。
一句话： FlashAttention 不是“用了 cache”，而是算法层面主动决定哪些张量必须落 HBM、哪些只在 SRAM 中短暂存在。

二、标准 Attention 做了什么，又慢在哪里

Self-attention 的数学形式大家很熟悉。设序列长度 N、head 维度 d： 

S = QK^T          # (N × N) 注意力分数
P = softmax(S)    # (N × N) 注意力权重
O = PV            # (N × d) 输出

当 N = 32k 时，S 和 P 各约 10 亿个元素。以 FP16 计，单个 N×N 矩阵就接近 2 GB。 更致命的是 HBM 往返次数：标准实现把 S 写回 HBM，再读出来做 softmax，再把 P 写回 HBM，再读出来乘 V。

论文中的一个关键观察：attention 运行时，softmax、mask、dropout 等 elementwise 操作占了大头时间， 尽管 FLOPS 主要集中在 MatMul。这类操作的算术强度（每字节内存访问对应的运算量）很低—— 属于 memory-bound，不是 compute-bound。

编译器社区把「把多个 GPU 操作合并成一次 HBM 读写」叫做 kernel fusion。 「kernel」在这里指 GPU 上的一个计算核；「materialization」指把中间结果完整分配并写入内存—— 标准 attention 物化了 S 和 P 两个 N×N 矩阵，这正是 FlashAttention 要消灭的。

三、FlashAttention 的两个核心思想

算法层面，FlashAttention 可以浓缩为两件事：

  • Tiling（分块）——前向和反向都用。把 N×N 的 softmax/score 矩阵切成小块，在 SRAM 内完成局部计算。
  • Recomputation（重算）——反向专用。不保存 S、P，反向时从 Q/K/V 和 softmax 统计量重新算。

四、分块的最大障碍：Softmax 需要「看到整行」

矩阵乘法可以分块，因为加法满足结合律：把大矩阵拆成小块分别乘再相加，结果不变。 但 softmax 不行——第 i 个元素的 softmax 分母需要该行所有 score： 

softmax(z_i) = exp(z_i) / Σ_j exp(z_j)

分块时，一次只能看到一行中的 B 个 score，分母还不完整。 如果强行按局部块做 softmax 再拼接，结果会错。

突破口是 online softmax（也称 streaming softmax）。 它维护两个 running statistics：

  • m：当前已见 score 的行最大值（数值稳定用）
  • l：归一化后的 exp 之和（softmax 分母的分子部分）

每处理一个新 block，用更新公式合并旧统计量与新 block 的统计量，输出 O 也同步 rescale。 递归处理完所有 block 后，得到与标准 softmax bit-exact 相同 的结果。

m(x) 的存在纯粹为了数值稳定（减最大值再 exp），不改变最终数学结果。 在假想的高精度硬件上，甚至可以省略——但在 FP16/BF16 的 GPU 上必不可少。

五、前向 Pass 算法：嵌套循环 + 分块

下面用单 head、batch=1、忽略 mask/dropout 的简化版说明。 记 B_r 为 Q 的行块大小，B_c 为 K/V 的列块大小，d 为 head 维度。

算法步骤概览：

1. 在 HBM 中分配 Q, K, V, O；额外分配 softmax 统计量 m, l（各 N 维）
2. 按 SRAM 容量计算块大小 B_r, B_c（约 ∝ M/4d，M 为 SRAM 容量）
3. 初始化 O=0，l=0，m=-∞
4. 外层 for j：加载 K_j, V_j 到 SRAM
5.   内层 for i：加载 Q_i, O_i, m_i, l_i 到 SRAM
6.     S_ij = Q_i @ K_j^T                    # 片上 B_r×B_c
7.     计算 block 内 m̃, l̃, P̃
8.     用 online softmax 公式更新 m_new, l_new, O_i
9.     写回 m_i, l_i, O_i
10. 循环结束 → O 即为最终 attention 输出

第 8 步是整个算法的心脏：它同时 rescale 之前累积的 O_i，并把当前 block 的贡献加进去。 那些 em−m_new 因子看起来复杂，本质是在「取消旧归一化、应用新归一化」—— 手算两三个 block 的 toy example，5 分钟就能建立直觉（Medium 原文 Aleksa Gordić 的 walkthrough 非常推荐）。

六、复杂度：从 O(N²) 显存到 O(N)

空间：标准 attention 需要 O(N²) 存 S 和 P；FlashAttention 只需 O(N·d) 存 Q/K/V/O， 外加 O(N) 存 m 和 l。当 N=64k、d=128 时，这是能否训练/推理长上下文的分水岭。

时间：渐近 FLOPs 仍是 O(N²)，但 wall-clock 时间看的是 HBM 访问次数。 标准实现 O(N²) 次 HBM 读写中间矩阵；FlashAttention 通过分块把 HBM 访问降到 O(N²·d²/M)， M 为 SRAM 容量。典型配置下可达 9× 更少 HBM 访问——这才是实际加速的来源。

关键洞察： 减少 FLOPs ≠ 更快。Attention 是 memory-bound；优化 IO 比优化算术量更能缩短 wall-clock 时间。

七、反向 Pass：Recomputation 而非 Checkpoint

训练时需要 backward。标准做法在 forward 保存 S、P 供 backward 使用——又是 O(N²) 显存。 FlashAttention 的做法类似 gradient checkpointing：forward 只存 O 和 (m, l)，backward 时从 Q/K/V 分块重算 S、P。

与普通 checkpointing 不同，FlashAttention 的重算仍在 SRAM 分块内进行， HBM 访问模式与前向一致——因此既省显存，又不显著拖慢 backward。 这是 IO-aware 重算，不是简单的「用算力换内存」。

八、从单 head 到生产：并行与扩展

上面的伪代码对应单个 CUDA thread block（跑在一个 SM 上）。 扩展到 batch × num_heads 时，每个 (batch, head) 对启动一个 thread block，并行跑在不同 SM 上。 A100 有 108 个 SM——越接近 SM 数量 × 并发度，利用率越高。

同一思路可扩展到 FlashAttention-2 的 work partitioning 优化， 以及 block-sparse FlashAttention：对 block 稀疏 mask 跳过部分 (i,j) 迭代， 稀疏度为 s 时可再快 ~1/s。

九、FlashAttention V1 → V4 演进脉络

版本 核心贡献 解决什么问题
V1 (2022) IO-aware exact attention；tiling + online softmax + recomputation N×N 中间矩阵 O(N²) 显存；HBM 带宽瓶颈
V2 (2023) 更好的 work partitioning；减少 rescale/同步；提高 Tensor Core 利用率 V1 在某些 shape 下 GPU 并行度不足、算力吃不满
V3 (2024) 面向 Hopper：异步 TMA、warp-specialization、流水化 load/compute 新架构上继续逼近 roofline 峰值
V4 (2025) 面向 Blackwell 及更新架构；新 Tensor Core 调度与内存管线 更大上下文、更多 attention 变体、新硬件代际

可以这么记：V1 改算法，V2 改并行，V3/V4 改硬件适配。 思想一脉相承——都是让数据尽量留在快内存里，少碰 HBM。

十、工程现实：为什么还需要 Triton / CUDA 专家

FlashAttention 的力量来自手写 CUDA kernel，这也意味着代价： 每个 attention 变体（不同 mask、不同 head dim、不同 GPU 架构）可能需要新 kernel； 代码难维护、难跨架构移植。

这正是 Triton、CUTLASS、FlashInfer 等项目的价值—— 在 Python/DSL 层写算法，编译器生成接近手写 CUDA 性能的 IO-aware kernel。 PyTorch 2.x 的 torch.nn.functional.scaled_dot_product_attention 在支持的硬件上也会自动 dispatch 到 FlashAttention 后端。

结论：FlashAttention 改变了什么，又没改变什么

没改变：attention 的数学定义、输出精度（exact，非近似）、O(N²) 的渐近 FLOPs。

改变了：中间矩阵是否 materialize 到 HBM；HBM 访问次数；可支持的序列长度上限； 训练/推理的 wall-clock 时间。

如果把 Transformer 推理看成一条数据流水线，FlashAttention 做的就是在最热的算子——attention——上， 重新设计「什么数据必须过 HBM、什么数据可以在 SRAM 里生灭」。 这不是 cache 优化技巧，而是算法与硬件存储层级 co-design 的范例。

算力可以指数增长，但 HBM 带宽追不上。FlashAttention 的回答是：别和硬件赛跑，改算法让它少跑 HBM。
