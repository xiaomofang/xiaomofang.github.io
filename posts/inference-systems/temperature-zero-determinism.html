temperature=0 时，大模型输出一定可复现吗？

一个常见判断是：temperature=0 等价于贪婪解码，因此同一 prompt 的两次输出必然一致。 这个判断只说对了一半。它描述的是采样策略的确定性，却没有覆盖推理系统的数值确定性。

在大模型应用开发中，temperature 往往被当作“随机性旋钮”：调高一些，输出更发散；调低一些，输出更保守； 设为 0，则每一步选择概率最大的 token。这个解释在解码层面没有问题，但它隐含了一个更强的前提： 每次前向计算得到的 logits 必须完全一致。现实中的线上推理服务并不总能满足这个前提。

Thinking Machines Lab 曾以 Qwen3-8B 为对象做过实验：在同一个 prompt、temperature=0 的条件下重复请求 1000 次， 最终得到 80 种不同回答。更重要的是，差异并非从第一步就出现；实验中前 102 个 token 完全一致，直到第 103 个 token 才发生分叉。 这类现象说明，问题不在“argmax 是否随机”，而在“参与 argmax 的 logits 是否逐 bit 稳定”。

一、temperature=0 固定的是解码策略

temperature 的作用发生在 logits 转换为采样分布之后。temperature 越高，分布越平，低概率 token 更容易被采到； temperature 越低，分布越尖锐，高概率 token 的优势越明显。当 temperature 被设为 0 时，实现上通常会直接采用贪婪解码， 即每一步取当前 logits 最大的 token。

这一步本身是确定的：如果 logits 完全相同，argmax 的结果也应完全相同。问题在于，线上推理不是一个孤立的数学函数调用， 它包含模型权重、CUDA kernel、batch 调度、缓存策略、并行通信和服务端版本等多个变量。temperature=0 只约束了解码层， 并没有约束这些系统变量。

关键区别： temperature=0 可以消除 sampling randomness，但不能自动消除 numerical nondeterminism。

二、真正的根因是 batch invariance 缺失

“GPU 并行导致浮点误差”是一个过于粗略的解释。浮点加法确实不满足结合律，例如在有限精度下， (1e30 + -1e30) + 1 与 1e30 + (-1e30 + 1) 可能得到不同结果。 但现代推理中的矩阵乘 kernel 通常不会依赖不受控的原子加来产生输出。对于同一输入、同一 kernel、同一执行形状， 单次算子本身往往可以保持 bitwise deterministic。

更准确的表述是：很多高性能 kernel 不具备 batch invariance。也就是说，同一个样本单独计算，和它被拼入更大 batch 后计算， 即使输入行本身完全相同，也可能走不同的分块、归约和合并路径。矩阵乘法、attention、RMSNorm 等算子都包含 reduction； reduction 的切分方式一旦随 batch 形状变化，浮点累加顺序也会随之变化。

在线服务恰好会不断改变 batch 形状。服务端为了吞吐，会把不同用户、不同长度、不同到达时间的请求动态合批。 从单个用户视角看，同一请求被放入哪个 batch、batch 中还有哪些 token、是否触发 chunked prefill 或缓存命中，都是不可控变量。 因此，服务器负载可以通过 batch 形状间接改变 logits 的最后几位。当 top-1 和 top-2 的差距足够小时，极小的数值差异就可能改变 argmax。

三、为什么顶尖模型开始收紧采样参数

近年来，部分头部闭源模型开始限制或移除 temperature、top_p、top_k 等参数。表面上看，这是 API 能力变少了； 从推理系统角度看，这是模型提供方将随机性控制从用户侧参数，逐步收回到模型与服务策略内部。

第一类原因是模型保护与蒸馏风险。采样参数越开放，外部用户越容易系统性观察模型在不同不确定性水平下的行为边界： 低温获取稳定答案，高温扩大候选覆盖，再通过大量查询构造训练数据。对闭源商业模型而言，减少采样自由度可以降低模型行为被高分辨率探测的风险。

第二类原因是推理链路复杂化。现代推理模型的输出未必来自一次简单的 next-token sampling，背后可能包含多轮候选生成、验证、重排、拒绝采样、 工具调用和内部策略控制。模型在训练和强化学习阶段通常已围绕默认推理配置做过优化。此时外部强行修改 temperature 或 top_p， 不一定带来更好的可控性，反而可能破坏模型被优化过的推理路径，导致循环、退化或复杂任务性能下降。

因此，“顶尖模型不再推荐手动调 temperature”与“temperature=0 不保证复现”并不矛盾。 前者说明采样控制正在系统化，后者说明确定性本来就不是单个参数能够提供的承诺。

四、需要复现时应如何设计系统

对闭源 API 而言，seed 通常只能提供 best-effort determinism。OpenAI Cookbook 提醒开发者同时记录 system_fingerprint， 用于感知后端权重、基础设施或数值配置的变化。Azure 文档也指出，即使 seed 和 fingerprint 一致，长输出仍可能出现差异； max_tokens 越大，生成过程中遇到临界 token 的概率越高，复现率自然下降。

  • 普通业务：固定模型版本、prompt、seed、temperature、top_p、max_tokens 等参数，并记录服务端 fingerprint。
  • 评测回归：不要只比较最终文本，应同时保存 token 序列、logprob、请求参数、模型版本和评测数据版本。
  • 审计场景：将输出复现性写入系统需求，明确允许的漂移范围，而不是默认 temperature=0 即可满足。
  • 强确定性场景：自建推理链路，采用 batch-invariant kernel，并接受一定吞吐损失以换取 bitwise 复现。

从工程实现看，真正的 bitwise 复现需要让关键 reduction 算子在不同 batch 形状下保持相同数值路径。 对 Transformer 推理而言，重点通常是 RMSNorm、矩阵乘法和 attention。Thinking Machines 的 batch-invariant ops 以及 SGLang 后续对确定性推理的集成， 都说明这不是概念问题，而是可被工程化约束的系统问题。

五、为什么这会影响 RL 训练

如果只是面向用户的聊天应用，偶发的措辞差异可能可以接受。但在 on-policy RL 中，数值不一致会进入训练假设本身。 典型 RLHF 或 RLAIF 系统中，sampler 通常使用 vLLM、SGLang 等推理引擎生成轨迹，trainer 则使用 FSDP、Megatron 等训练框架计算梯度。 两套系统的 kernel、并行方式和数值路径并不天然一致。

理论上，sampler 采样的数据应来自当前被优化的 policy；这正是 on-policy 的前提。若 sampler 与 trainer 的 logits 因数值路径不同而系统性偏移， 采样策略与更新策略之间就会出现 KL 漂移，训练过程会在不自知的情况下引入 off-policy 偏差。Thinking Machines 的实验显示， batch-invariant kernel 可以使 sampler 与 trainer 之间的 KL 保持为 0；普通非确定 kernel 则可能在训练中持续积累偏移，最终触发 reward collapse。

结论：确定性是一项系统属性

temperature=0 的正确边界是：它将解码策略固定为 argmax，但不保证 logits 计算过程逐 bit 一致。 logits 的微小差异可能来自浮点归约顺序、kernel 实现、batch 形状、服务端调度、MoE 路由、并行通信或后端版本变化。 当这些因素共同作用时，同一个 prompt 的两次输出出现差异并不是反常现象，而是高性能推理系统的自然结果。

因此，这道问题真正考察的不是 temperature 的定义，而是对推理系统边界的理解： 参数可以约束解码策略，seed 可以提高复现概率，fingerprint 可以帮助定位后端变化； 但严格确定性需要从 kernel、batching、服务调度和评测设计层面共同建立。

temperature=0 不是确定性的终点，而是追问确定性从哪一层开始失效的起点。
